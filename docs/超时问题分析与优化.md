# 知识库超时问题分析与优化方案

## 问题概述

知识库查询出现响应超时,系统无法在规定时间内返回结果。

## 诊断结果

### 1. 系统组件状态

| 组件 | 状态 | 性能 |
|------|------|------|
| Milvus向量库 | ✅ 正常 | 搜索耗时 0.52s |
| LightRAG图谱 | ✅ 正常 | 查询耗时 ~66s ⚠️ |
| MySQL数据库 | ✅ 正常 | - |
| Redis缓存 | ✅ 正常 | - |

### 2. 超时原因定位

**主要瓶颈: LightRAG图谱检索耗时过长**

完整查询流程耗时分析:
```
总耗时: 71.64秒
├─ 查询改写: 2.8s
├─ 意图识别: 1.9s
├─ LightRAG检索: ~66s ⚠️ 瓶颈
│  ├─ 实体向量检索
│  ├─ 关系向量检索
│  ├─ 图谱遍历
│  └─ LLM生成答案: ~66s
└─ Milvus检索: 0.5s
```

**触发超时的配置冲突:**
- 前端超时: 60秒 (`web/src/api/request.js:6`)
- 实际查询: 71.64秒
- **结果: 超时!**

### 3. 为什么LightRAG较慢?

LightRAG的查询过程包含:
1. **Hybrid模式** 同时执行local + global检索
2. 查询了 **72个实体 + 159个关系**
3. 提取了 **25个文本片段**
4. 最终生成 **17个上下文块**
5. **LLM总结** 大量上下文(耗时最长)

### 4. 数据库连接正常

## 优化方案

### ✅ 方案1: 立即生效 - 调整超时配置

**已修改文件:**
1. **web/src/api/request.js**
   ```javascript
   // 修改前
   timeout: 60000  // 60秒

   // 修改后
   timeout: 120000  // 120秒 - 适配LightRAG慢查询
   ```

2. **.env配置**
   ```bash
   # 修改前
   PERFORMANCE__QUERY_TIMEOUT=30

   # 修改后
   PERFORMANCE__QUERY_TIMEOUT=90  # 给后端更多处理时间
   ```

**效果:**
- 前端等待时间: 60s → 120s
- 后端查询超时: 30s → 90s
- 能够处理慢查询,不再超时

### 🚀 方案2: 长期优化 - 提升LightRAG性能

#### 2.1 减少检索数据量

编辑 `app/agents/pipeline/graph_agent.py:44`:
```python
# 当前配置
result = await self._graph_agent.process_request(
    {"query_text": query, "mode": "hybrid", "top_k": 3},
    MessageContext(),
)

# 优化建议
result = await self._graph_agent.process_request(
    {"query_text": query,
     "mode": "local",  # 简单查询用local,复杂查询用hybrid
     "top_k": 2,       # 减少返回数量
     "only_need_context": False
    },
    MessageContext(),
)
```

#### 2.2 启用结果缓存

LightRAG已自动缓存LLM响应:
```
INFO:  == LLM cache == saving: hybrid:query:xxx
```

可在 `.env` 中增加:
```bash
# LightRAG缓存配置
LIGHTRAG_ENABLE_LLMCACHE=true
LIGHTRAG_CACHE_DIR=resources/data/lightrag/cache
```

#### 2.3 优化查询策略

修改 `app/agents/service/agent_service.py:274` 的路由逻辑:

```python
def _resolve_route(self, intent_result, rewritten_query, ...):
    # 当前: 所有process类问题都走graph
    if sub_intent == "process":
        return "graph"

    # 优化: 简单process问题走knowledge(快),复杂的才走graph
    if sub_intent == "process":
        if self._is_simple_process_query(rewritten_query):
            return "knowledge"  # Milvus检索,0.5秒
        return "graph"  # 复杂关系,66秒
```

#### 2.4 异步+流式响应(推荐)

将慢查询改为流式返回,提升用户体验:

```python
async def answer_streaming(self, query):
    """流式返回LightRAG结果"""
    # 1. 快速返回初步答案(来自Milvus)
    yield {"status": "partial", "answer": quick_answer}

    # 2. 后台继续LightRAG检索
    graph_answer = await self.graph_agent.answer(query)

    # 3. 追加详细答案
    yield {"status": "complete", "answer": graph_answer}
```

### ⚡ 方案3: 智能路由优化

根据查询复杂度动态选择:
- **简单查询** → Milvus向量检索 (0.5秒)
- **关系查询** → LightRAG图谱 (60秒)
- **混合查询** → 并行检索 + 结果融合
