## 核心策略

### 两阶段切块流程

```
原始文档
    ↓
【第一阶段】滑动窗口初步切分
    - Chunk Size: 1000 tokens
    - Overlap: 100 tokens
    ↓
【第二阶段】关键词感知智能切分
    - 检测关键词位置
    - 在关键词边界切分
    - 保留关键词上下文
    ↓
最终Chunks
```

## 关键词感知切分算法

### 核心逻辑

```python
def _split_by_keywords(self, text: str) -> List[str]:
    """
    1. 查找所有关键词位置
    2. 确定包含关键词的chunk范围
       - 起点: 关键词前 keyword_context_window 个字符
       - 终点: 关键词后 keyword_context_window 个字符
    3. 尝试合并相邻的关键词（如果距离 < context_window）
    4. 在句子边界结束（句号、换行等）
    5. 确保chunk不超过 max_chunk_size
    """
```

### 优势

1. **完整语义保留**
   - ✅ 不会在关键词中间切断
   - ✅ 保留关键词前后80字符上下文
   - ✅ 在句子边界切分

2. **智能合并**
   - ✅ 相邻的多个关键词会被包含在同一chunk
   - ✅ 避免产生过多碎片化的小chunk

3. **灵活控制**
   - ✅ 最小/最大chunk限制
   - ✅ 可调整的上下文窗口
   - ✅ 自动处理剩余文本

| 方法 | Chunk数 | 平均大小 | 关键词保留 | 语义完整性 |
|------|---------|---------|-----------|-----------|
| **固定大小切分** | 2-3个 | ~300 tokens | ❌ 可能切断 | ⚠️ 中等 |

## 使用场景

### 适用场景 ✅

1. **政策文档**
   - 包含大量专业术语和关键词
   - 需要保留完整的条款上下文
   - 本测试案例即为典型应用

2. **技术文档**
   - API名称、函数名等关键词
   - 代码示例不应被切断
   - 技术术语需要完整上下文

3. **法律文档**
   - 法律条文、术语
   - 引用和定义需要完整
   - 上下文对理解至关重要

4. **领域知识库**
   - 行业术语丰富
   - 关键概念需要精确检索
   - 上下文依赖性强

### 不适用场景 ⚠️

1. **叙事性文本**
   - 小说、故事等
   - 关键词不明确
   - 更适合语义切分

2. **对话/聊天记录**
   - 碎片化内容
   - 无明确关键词
   - 更适合按轮次切分

3. **多媒体内容描述**
   - 图片说明、视频字幕
   - 内容过于简短
   - 不需要复杂切分

## 改进建议

### 当前版本已很好 ✅

测试结果表明算法按预期工作，无重大缺陷。

### 可选优化方向

1. **关键词权重**
   ```python
   keywords = {
       "补贴": 1.0,        # 核心词，高权重
       "第一章": 0.5,      # 结构词，低权重
   }
   ```
   - 根据关键词重要性调整切分策略
   - 优先保留高权重关键词的上下文

2. **动态上下文窗口**
   ```python
   if keyword_importance > 0.8:
       context_window = 120  # 重要关键词，更大上下文
   else:
       context_window = 50   # 次要关键词，标准上下文
   ```

3. **层次化关键词**
   ```python
   keywords = {
       "章节": ["第一章", "第二章", ...],
       "核心": ["补贴", "能效标准", ...],
       "细节": ["申请条件", "发票", ...]
   }
   ```
   - 按层次组织关键词
   - 不同层次不同切分策略

4. **重叠关键词处理**
   - 当前: 简单合并相邻关键词
   - 改进: 智能判断是否应该在关键词簇之间切分
