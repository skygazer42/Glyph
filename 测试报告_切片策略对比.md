## 切片策略对比

### 1️⃣ Token切片 - 固定Token数量

#### 配置 A: 无重叠
```python
TokenTextSplitter(
    chunk_size=30,
    chunk_overlap=0
)
```

**结果**:
- ✅ 切片数量: **41个**
- ⚠️ 平均长度: 26.7字符
- ❌ 问题:
  - 切片过于碎片化
  - 频繁切断句子
  - 语义不完整

**示例**:
```
切片 3: "具，可以帮助开发者将私有数据与大型语言模型（LLMs"
切片 4: "）连接起来，"
切片 7: "其核心概念包括文档加载器、节点解析器、索"
切片 8: "引和查询引擎。"
```

**问题分析**:
- "LLMs" 被切断成 "LLMs" 和 "）连接起来"
- "索引" 被分成 "索" 和 "引"
- 严重破坏了语义完整性

---

#### 配置 B: 有重叠
```python
TokenTextSplitter(
    chunk_size=30,
    chunk_overlap=10
)
```

**结果**:
- ✅ 切片数量: **54个** (增加了13个，因为重叠)
- ⚠️ 平均长度: 20.3字符
- ❌ 问题:
  - 更加碎片化
  - 重叠没有解决断句问题
  - 增加了存储和计算成本

**示例**:
```
切片 1: "LlamaIndex 是一个用于构建 LLM"
切片 2: "是一个用于构建 LLM 应用程序的数据框架。"
切片 3: "它提供了一套工具，可以帮助开发者"
```

**重叠效果**: 虽然有10 token重叠，但仍然切断了句子，效果不佳。

---

### 对比总结: Token切片

| 指标 | 无重叠 | 有重叠 |
|------|--------|--------|
| 切片数量 | 41 | 54 (+32%) |
| 平均长度 | 26.7字符 | 20.3字符 |
| 语义完整性 | ❌ 差 | ❌ 差 |
| 存储效率 | ✓ 较好 | ⚠️ 中等 |
| 适用场景 | ❌ 不推荐 | ❌ 不推荐 |

**结论**: Token切片在chunk_size太小时(30 tokens)严重破坏语义，**不适合中文文档**。

### 3️⃣ SentenceWindow切片 - 带上下文窗口

```python
SentenceWindowNodeParser(
    window_size=3,      # 前后各3个句子
    window_metadata_key="window",
    original_text_metadata_key="original_text"
)
```

**结果**:
- ✅ 切片数量: **1个** (文档较短，被完整包含)
- ✅ 长度: 1098字符 (整个文档)
- ⚠️ 说明: 窗口大小(3句)包含了整个文档

**工作原理**:
```
核心句子: [第一段内容]
  ↓
窗口上下文: 核心句子 + 前3句 + 后3句
  ↓
由于文档较短，窗口覆盖了整个文档
```

**元数据**:
- `window`: 完整的窗口内容（用于LLM）
- `original_text`: 核心句子（用于索引/检索）

**适用场景**:
- ✅ 长文档切分后需要保留上下文
- ✅ 问答系统需要前后文信息
- ⚠️ 对于短文档，窗口可能包含整个文档

**注意**: 在实际RAG应用中，检索时返回核心句子，但LLM看到的是完整窗口内容。

### 4️⃣ Semantic切片 - 语义驱动

```python
SemanticSplitterNodeParser(
    buffer_size=1,
    breakpoint_percentile_threshold=95,
    embed_model=Settings.embed_model
)
```

**测试状态**: ⚠️ 跳过（需要DASHSCOPE_API_KEY配置）

**工作原理**:
1. 计算相邻句子之间的语义相似度
2. 识别语义突变点（相似度低于阈值）
3. 在语义边界处切分

**优势**:
- ✅ 智能识别主题转换
- ✅ 保持语义连贯性
- ✅ 适应不同文档结构

**成本**:
- ❌ 需要调用embedding API
- ❌ 计算开销较大
- ❌ 速度较慢

## 实际应用建议

### 政策文档RAG系统（当前项目）

**推荐方案 1**: Sentence切片 + 关键词感知
```python
# 第一阶段: Sentence切片
sentence_splitter = SentenceSplitter(
    chunk_size=800,
    chunk_overlap=100
)

# 第二阶段: 关键词感知二次切分
keyword_aware_parser = KeywordAwareParser(
    primary_parser=sentence_splitter,
    keywords=policy_keywords,
    max_chunk_size=600
)
```
**优势**:
- ✅ 尊重句子边界
- ✅ 保留政策关键词完整性
- ✅ 性能优秀（无需embedding）

**推荐方案 2**: Sentence切片 + Reranker
```python
# 切片
sentence_splitter = SentenceSplitter(
    chunk_size=600,
    chunk_overlap=80
)

# 检索 + Rerank
retriever → reranker → top_k results
```
**优势**:
- ✅ 简单高效
- ✅ Reranker弥补切片不足
- ✅ 已验证效果好（测试报告3）

**不推荐**: Token切片（会破坏中文语义）

## 关键发现

### 1. Token切片的问题

在**中文文档**和**小chunk_size**下，Token切片会：
- ❌ 切断单词/汉字
- ❌ 破坏语义
- ❌ 降低检索质量

**示例**:
```
"其核心概念包括文档加载器、节点解析器、索引和查询引擎。"
                                        ↓
Token切片(30): "其核心概念包括文档加载器、节点解析器、索"
               "引和查询引擎。"
```

### 2. Sentence切片是通用最佳实践

**原因**:
1. ✅ 尊重自然语言边界
2. ✅ 语义完整性好
3. ✅ 性能优秀
4. ✅ 无额外API成本
5. ✅ 易于调试和理解

**配置建议**:
- 短文档: chunk_size=200-400
- 中等文档: chunk_size=400-800
- 长文档: chunk_size=800-1200

### 3. SentenceWindow适合特定场景

**何时使用**:
- ✓ 问答系统需要上下文
- ✓ 文档较长(>5000 tokens)
- ✓ 复杂推理任务

**何时不用**:
- ✗ 短文档(<1000 tokens)
- ✗ 简单关键词检索
- ✗ 存储空间紧张

### 4. 混合策略是高级方案

**示例**: Sentence + KeywordAware
```
第一阶段: Sentence切片
    ↓
第二阶段: 检测关键词，在大chunk中二次切分
    ↓
最终: 大小适中 + 语义完整 + 关键词保留
```

**已实现**: `test_keyword_chunking.py`（测试报告4）

---

## 最佳实践总结

### 通用规则

1. **优先使用Sentence切片**
   - 适合90%的场景
   - 平衡了质量和性能

2. **chunk_size设置**
   ```
   经验值:
   - 最小: 200 tokens (避免过于碎片)
   - 推荐: 500-800 tokens (平衡)
   - 最大: 1500 tokens (避免过长)
   ```

3. **overlap设置**
   ```
   经验值:
   - chunk_size的10-20%
   - 例如: chunk_size=500 → overlap=50-100
   ```

4. **特殊场景**
   - 代码文档: TokenTextSplitter (chunk_size > 500)
   - 问答系统: SentenceWindowNodeParser
   - 多主题文档: SemanticSplitterNodeParser
