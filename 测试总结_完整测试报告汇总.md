### 4️⃣ [关键词感知切块测试](./测试报告_关键词感知切块.md)
**测试脚本**: `test_keyword_chunking.py`
**测试内容**: 滑动窗口 + 关键词感知的混合切块

#### 切块策略

```
原始文档 (815字符)
    ↓
【第一阶段】滑动窗口 (1000 tokens, 100 overlap)
    ↓ 输出: 1个大chunk (869 tokens)
【第二阶段】关键词感知切分
    - 检测84个关键词位置
    - 在关键词边界智能切分
    - 保留关键词上下文 (±80字符)
    ↓
最终输出: 2个chunk (571, 298 tokens)
```

#### 切块参数
| 参数 | 值 | 说明 |
|------|---|------|
| chunk_size | 1000 tokens | 第一阶段窗口大小 |
| max_chunk_size | 500 字符 | 触发二次切分阈值 |
| keyword_context_window | 80 字符 | 关键词上下文保留 |
| 关键词数量 | 40个 | 政策领域关键词 |

#### 切块效果

**Chunk 1.1** (571 tokens)

- 内容: 标题 → 第一章 → 第二章 → 申请条件前半部分
- 边界: 在条款中间切分 ✓

**Chunk 1.2** (298 tokens)
- 包含15个关键词: 能效标准、第四章、第五章、市场监管...
- 内容: 申请条件后半部分 → 监督管理 → 附则
- 边界: 保持流程连贯性 ✓

#### 质量指标
- ✅ **关键词覆盖率**: 100%
- ✅ **语义完整性**: 优秀
- ✅ **边界质量**: 尊重文档结构

## 系统架构总结

### 完整RAG流程

```
[1] 文档处理
    ├── 读取政策文档 (MD格式)
    ├── 层次化解析 (Document → Section → Chunk)
    ├── Section: LLM摘要生成 ✓
    └── Chunk: 关键词感知切分 ⭐

[2] 索引构建
    ├── 向量嵌入 (DashScope text-embedding-v3)
    ├── 存储到Milvus
    └── 构建层次关系 (PARENT/CHILD)

[3] 查询处理
    ├── 用户问题
    ├── 关键词检索 (或向量检索)
    ├── 召回候选 (Top 20)
    └── Reranker精排 (Top 5) ⭐

[4] 答案生成
    ├── 选取Top 3文档
    ├── 组合上下文 (限制4000字符)
    └── LLM生成答案
```

### 核心组件

| 组件 | 技术选型 | 状态 |

| **Reranker** | gte-rerank-v2 (DashScope) | ✅ 高效可用 |
| **向量数据库** | Milvus | ✅ 已启动 |

## 关键发现

### 1. 多级索引的价值

**Section级别** (LLM摘要):
- ✅ 适合快速浏览和概览
- ✅ 帮助用户快速定位相关章节
- ✅ 摘要质量高，信息密度大

**Chunk级别** (原文):
- ✅ 保留完整细节
- ✅ 适合精确检索
- ✅ 支持引用和验证

### 2. Reranker是必需的

**不使用Reranker**:
- 检索时间: 0.001秒
- 总响应时间: 1.242秒
- 准确性: 中等 ⚠️

**使用Reranker**:
- 检索时间: 0.001 + 0.260 = 0.261秒
- 总响应时间: 1.467秒 (+18%)
- 准确性: 高 ✅

**结论**: 仅增加18%时间，显著提升准确性，非常值得！

### 3. 关键词感知切块的优势

传统方法问题:
- ❌ 可能在关键词中间切断
- ❌ 不保留关键词上下文
- ❌ 语义完整性差

关键词感知方案:
- ✅ 保证关键词完整
- ✅ 保留±80字符上下文
- ✅ 100%关键词覆盖
- ✅ 无需LLM，性能优秀

### 4. 当前架构是 Reranker + 多级索引

```
检索时使用 Reranker:
  用户查询 → 向量检索 → 召回候选 → Reranker排序 → 返回Top N

索引时使用多级结构:
  文档 → Section(LLM摘要) → Chunk(原文) → 存入Milvus
```

**两者独立且互补**:
- 多级索引: 数据组织方式
- Reranker: 检索结果排序
